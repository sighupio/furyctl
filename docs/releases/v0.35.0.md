# furyctl release v0.35.0

Welcome to the latest release of `furyctl` maintained by SIGHUP by ReeVo team.

## New features 🌟

- [[#658](https://github.com/sighupio/furyctl/pull/658)] Introduced new SIGHUP Distribution `Immutable` kind in alpha status, capable of creating SD clusters with Flatcar Container Linux based nodes.
- Add compatibility with **SIGHUP Distribution v1.35** and widen the latest 1.33/1.34 ranges to the next patch (`v1.33.3`, `v1.34.2`).
- [[#672](https://github.com/sighupio/furyctl/pull/672)] `furyctl` now downloads and validates only the dependencies each cluster kind actually needs (provider tool sections, the single installer for the kind, and the provider modules). Backward compatible with older distributions: tools pinned under `tools.common` (and the legacy `terraform`) are still resolved, with the provider section (`tools.eks`) taking precedence.
- [[#673](https://github.com/sighupio/furyctl/pull/673)] `furyctl` now installs all downloadable tools through a bundled, checksum-pinned `mise` instead of the bespoke per-tool downloader, and can build/consume a self-contained **air-gapped bundle** to run fully offline.
- [[#674](https://github.com/sighupio/furyctl/pull/674)] `furyctl` now **owns its config types**: small, hand-maintained structs that model only the fields furyctl reads. This drops the build-time dependency on `fury-distribution`'s generated Go module (no more codegen / three-repo dance). Transparent for CLI users.
- [[#652](https://github.com/sighupio/furyctl/pull/652)] Added the `get cluster-info` command to display a quick overview of a running cluster's configuration and status.

## Bug fixes 🐞

- Copying a distribution from a local `--distro-location` no longer fails because of furyctl's own `.furyctl` working directory: that directory (which may contain tool shims/symlinks created by the bundled mise) is now excluded from the copy, and any dangling symlinks elsewhere are skipped instead of aborting the whole copy.
- EKSCluster: restored the `json` struct tags on the hand-maintained config types introduced in #674. Without them the Terraform/OpenTofu data injection (VPC id, IAM role ARNs, Route53 hosted zone) was keyed by Go field names instead of the camelCase schema keys, so the distribution phase rendered an empty `vpc_id` and `tofu plan` failed on every EKS cluster.
- [[#667](https://github.com/sighupio/furyctl/pull/667)] Fixed a panic that occurs when deploying an EKS cluster with spec.kubernetes.apiServer.publicAccess: false and using an existing VPC.
