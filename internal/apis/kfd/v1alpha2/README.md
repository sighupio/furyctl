# furyctl configuration types — where to act

This document explains where and how to change the Go types furyctl uses to read
a `furyctl.yaml`, now that **there is no more autogenerated library and no more
private schema** in `fury-distribution`.

## What changed

Previously the config types were **generated** from the JSON schemas in
`fury-distribution` by `go-jsonschema` (a SIGHUP fork), and furyctl imported them
as `github.com/sighupio/fury-distribution/pkg/apis/...`. For EKS, a JSON-patch
produced a "private" schema with extra internal fields, also code-generated.

Now:

- **furyctl owns its config types**, hand-written and curated, under
  `internal/apis/kfd/v1alpha2/<provider>/{public,private}/schema.go`.
- **No codegen, no `go-jsonschema`, no published types library.** furyctl no
  longer depends on the `fury-distribution` Go module at all.
- `fury-distribution` is now a **data-only repo**: it ships the JSON **public**
  schemas, defaults, templates, rules and `kfd.yaml` — no Go.

## Where the types live

| Type | Location | Notes |
|---|---|---|
| `kfddistribution` config | `internal/apis/kfd/v1alpha2/kfddistribution/public/schema.go` | |
| `onpremises` config | `internal/apis/kfd/v1alpha2/onpremises/public/schema.go` | |
| `immutable` config | `internal/apis/kfd/v1alpha2/immutable/public/schema.go` | |
| `ekscluster` config | `internal/apis/kfd/v1alpha2/ekscluster/private/schema.go` | see "EKS private" below |
| `kfd.yaml` + bootstrap `furyctl.yaml` (`Kind`/`apiVersion`) | `internal/apis/config/` | hand-written, moved from the old `fury-distribution/pkg/apis/config` |

These structs model **only the fields furyctl actually reads** (a small subset
of the full schema). They are meant to read like a `furyctl.yaml` skeleton.

## How to add or change a field furyctl reads

1. Open the curated struct for the relevant provider.
2. Add/adjust the field, mirroring the JSON-schema field name in its `yaml:"..."`
   tag. Copy the exact type from the schema (use the published schema in
   `fury-distribution/schemas/public/` as reference).
   - Optional fields → pointer (`*string`, `*int`, …) or slice.
   - Enums → keep a comment listing the allowed values.
   - For complex/repeated shapes use a named type (`Node`, `Member`, …); inline
     anonymous structs for simple nesting.
3. Use the field in furyctl code as usual (`conf.Spec...`).

You do **not** regenerate anything. There is no build step.

## Validation (important)

Field correctness of the user's input is **not** enforced by these structs. The
full `furyctl.yaml` is validated **at runtime** against the public JSON schema
downloaded from the distribution, in `internal/config.Validate`
(`internal/schema/santhosh`). The curated structs are furyctl's *read-view* only;
fields you omit are still validated by the schema and still reach the templates
via the raw YAML (`internal/cluster/phase.go` reads the file as a generic map).

## EKS "private" types

The `ekscluster/private` package is called "private" for historical reasons. It
models, besides the user-read fields, the **EKS-internal fields furyctl fills in
itself** by injecting infrastructure (Terraform/OpenTofu) outputs before
rendering templates — see `ekscluster/common/distribution.go`
(`injectDataPreTf` / `InjectDataPostTf`): VPC id and the IAM role ARNs
(`ebs_csi_driver`, `load_balancer_controller`, `cluster_autoscaler`,
`external_dns`, `cert_manager`, `velero`). These values are computed during the
run and are never user input — which is exactly why they live in furyctl now.

If you add a new injected field, add it both to the curated struct here and to
the injection payload in `common/distribution.go`.

## Avoiding drift from the schema

Since the types are hand-maintained, they can drift from the JSON schema. Two
safety nets:

- the **compiler** catches any field you read that you renamed/removed;
- an **anti-drift test** decodes reference `furyctl.yaml` examples into the
  curated structs and asserts the fields furyctl needs are present.

When the distribution schema changes a field furyctl reads, update the curated
struct accordingly.
